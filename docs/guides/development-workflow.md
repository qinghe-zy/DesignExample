# 开发与维护流程

## 文档目的

本文档说明当前仓库推荐的开发、维护与交付流程。

## 标准执行顺序

1. 检查当前仓库状态
2. 读取治理文件与过程记忆文件
3. 判断修改应属于：
   - `docs/`
   - `shared/`
   - `blueprints/`
   - 某个具体项目
4. 做最小但有价值的修改
5. 执行相应验证
6. 更新 Documentation、CHANGELOG、HANDOFF 等文档

## 何时修改哪一层

### 修改 `docs/`

当变更主要面向人类理解时，例如：

- 仓库结构说明
- 工作流程说明
- 模板创建指南
- 模板目录说明

### 修改 `shared/`

当变更属于跨项目共用内容时，例如：

- 命名规范
- 响应规范
- 权限规范
- 验证清单
- 派生项目推进清单

### 修改 `blueprints/`

当变更会影响“项目如何起步”时，例如：

- 基础蓝图结构
- 蓝图边界
- 扩展点定义

### 修改某个 `projects/<name>/`

当变更只属于具体项目实现时，例如：

- 项目模块代码
- 项目数据库脚本
- 项目 README
- 项目启动说明

## 当前验证基线

对于非平凡修改，尽量执行以下验证中的相关项：

- 后端构建
- 后端测试
- 前端安装
- 前端构建
- 数据库初始化
- MySQL profile 测试上下文
- 文档一致性检查

若因环境限制无法完成，应明确记录：

- 实际执行了什么命令
- 哪里失败
- 是代码问题还是环境问题
- 后续手动步骤是什么

## 复用判断规则

新增内容前，先问三个问题：

1. 它是否已经被蓝图覆盖？
2. 它是否很可能出现在多个项目里？
3. 它是否已经足够稳定，值得抽到共享层？

优先结果：

- 明显共性 -> 放入 `shared/`
- 明显起步结构 -> 放入 `blueprints/`
- 仍然业务化 -> 先留在项目内

## 当前文档维护要求

每完成一次重要修改，应至少更新：

- `Documentation.md`
- `CHANGELOG.md`
- `HANDOFF.md`

如果涉及关键工程取舍，还应更新：

- `DECISIONS.md`

## 当前交付标准

本仓库的交付不以“文件存在”为准，而以“状态可解释、可验证、可继续推进”为准。
