# 如何创建一个新模板

## 文档目的

本文档说明在当前模板工厂中，如何以一致、可维护、可扩展的方式创建新模板项目。

## 开始之前先看什么

创建新模板前，建议先阅读：

1. `docs/catalog/template-catalog.md`
2. `docs/architecture/workspace-architecture.md`
3. `shared/conventions/naming-rules.md`
4. `blueprints/base-admin-template/README.md`

原则始终是：**先复用，后特化**。

## 推荐创建流程

### 第一步：确认模板类别

先明确：

- 项目名称
- 目标用户
- 主要业务模块
- 哪些能力可直接继承基础底座
- 哪些能力仍需保留在项目本地

如果属于新类别，应先更新模板目录，而不是直接生成一个“未登记项目”。

### 第二步：选择起点

如果项目仍属于管理端系统，优先从 `blueprints/base-admin-template` 起步。

它默认提供：

- 登录
- RBAC
- 仪表盘
- 用户 / 角色 / 菜单
- 通用 CRUD 页面范式

### 第三步：初始化项目目录

项目目录统一放在 `projects/` 下，名称使用小写 kebab-case。

示例：

- `student-management`
- `library-management`
- `exam-system`

### 第四步：继承基础模块

新模板通常应保留以下基线能力：

- 登录 / 登出
- 当前用户
- 用户管理
- 角色管理
- 菜单管理
- 仪表盘
- 至少一个共享 CRUD 范式

### 第五步：逐步加入业务模块

不要一次性把所有业务模块全部灌进去。

建议顺序：

1. 先实现 1 个代表业务模块
2. 同步数据库与文档
3. 验证当前实现
4. 再继续扩展其余模块

### 第六步：记录可复用模式

如果项目内出现很可能在其他项目重复出现的模式，应记录为：

- 共享规范候选
- 蓝图增强候选
- 派生项目推进清单中的复用步骤

### 第七步：验证

一个新模板至少应尽量验证：

- 后端构建
- 后端测试
- 前端安装
- 前端构建
- 数据库初始化
- 启动说明是否真实可用

## 当前建议的项目状态分级

### 1. 强化脚手架状态

至少应具备：

- 项目 README
- 后端/前端 README
- 模块说明
- 扩展说明
- SQL/schema 基线
- 代表模块骨架

### 2. 可运行派生项目状态

除脚手架状态外，还应具备：

- 后端构建可通过
- 前端构建可通过
- 数据库初始化可执行
- 最好具备测试上下文验证

## 何时应新增蓝图而不是继续复用基础管理端蓝图

当多个项目持续朝同一方向偏离基础底座时，应考虑新蓝图，例如：

- 内容平台类蓝图
- 移动优先类蓝图
- 考试引擎类蓝图

这样可以避免把所有差异都硬塞回基础管理端底座中。
